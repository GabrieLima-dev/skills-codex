# Benchmark Visual para Frontend

Use esta referência para montar exatamente 3 propostas de direção visual com links clicáveis no plano obrigatório.

As propostas devem ser escolhidas conforme contexto do projeto. Não copie layout, assets, marcas, ilustrações, textos ou código. Extraia linguagem visual, estrutura e princípios de experiência.

## Como escolher as 3 propostas

1. Se o usuário já trouxe site, imagem ou vídeo de referência, uma proposta pode sintetizar essa referência.
2. As outras propostas devem ampliar o leque visual com alternativas realmente diferentes.
3. Cada proposta precisa ter link clicável e trade-off curto.
4. As 3 propostas devem ajudar o usuário a decidir direção, não apenas listar sites famosos.

## Template obrigatório da proposta

```md
### Proposta 1 - <nome da direção>
- Referência: [Nome do site](https://exemplo.com/)
- Quando usar: <contexto ideal>
- Linguagem visual: <layout, hierarquia, paleta, tipografia, componentes, motion>
- Trade-off: <ganho principal> / <risco ou custo de adaptação>
```

## Referências por tipo de interface

### Portfólio visual / experiência imersiva

- [samsy.ninja](https://samsy.ninja/)
  - Padrão: experiência full-canvas, UI mínima sobre cena visual, motion forte e atmosfera autoral.
  - Quando usar: portfólios criativos, páginas de impacto e narrativas visuais altamente customizadas.

- [Bruno Simon](https://bruno-simon.com/)
  - Padrão: interação lúdica, experiência 3D como navegação e identidade visual memorável.
  - Quando usar: portfólio experimental ou projeto onde a interação é parte do posicionamento.

- [Resn](https://resn.co.nz/)
  - Padrão: direção artística digital, composição não convencional e alto impacto visual.
  - Quando usar: experiências criativas, campanhas e sites onde diferenciação estética é prioridade.

### Produto premium / landing page

- [Apple](https://www.apple.com/)
  - Padrão: minimalismo premium, ritmo editorial, alto respiro, narrativa visual forte.
  - Quando usar: produto ou marca que precisa parecer precisa, elegante e aspiracional.

- [Linear](https://linear.app/)
  - Padrão: SaaS sofisticado, contraste refinado, microinterações discretas e forte clareza de produto.
  - Quando usar: ferramentas digitais, dashboards e páginas com foco em produtividade.

- [Vercel](https://vercel.com/)
  - Padrão: tecnologia premium, tipografia forte, seções concisas, contraste alto e grids limpos.
  - Quando usar: produto técnico, plataforma dev, infraestrutura ou SaaS moderno.

### Dashboard / ferramenta operacional

- [Stripe Docs](https://docs.stripe.com/)
  - Padrão: legibilidade técnica, navegação clara, blocos funcionais e hierarquia objetiva.
  - Quando usar: documentação, portais técnicos, backoffices e fluxos orientados a tarefas.

- [GitHub](https://github.com/features)
  - Padrão: interface densa, pragmática, com navegação familiar e estados claros.
  - Quando usar: produto operacional, ferramenta para usuários recorrentes e fluxos de produtividade.

- [Raycast](https://www.raycast.com/)
  - Padrão: UI escura refinada, foco em comandos, ícones, listas densas e interações rápidas.
  - Quando usar: ferramentas de produtividade, launchers, painéis e apps com workflows intensos.

## Critérios de avaliação por proposta

Para cada uma das 3 propostas, avalie:

- Layout: editorial, dashboard, full-canvas, split, grid, sidebar, command center.
- Hierarquia: qual elemento domina a primeira dobra e qual ação vem em seguida.
- Paleta: contraste, temperatura visual, saturação e risco de monotonia.
- Tipografia: escala, peso, legibilidade e personalidade.
- Espaçamento: densidade, ritmo e organização responsiva.
- Componentes: cards, tabelas, menus, botões, filtros, painéis, overlays.
- Motion: utilidade, intensidade, duração e risco de excesso.
- Acessibilidade: contraste, foco visível, navegação por teclado e textos legíveis.

## Regra operacional obrigatória

- Em qualquer mudança frontend visual/layout/theme/responsividade/UX: apresentar exatamente 3 propostas de direção visual com links clicáveis no plano obrigatório.
- Sempre incluir trade-offs curtos por proposta.
- Nunca implementar antes da escolha explícita do usuário.
